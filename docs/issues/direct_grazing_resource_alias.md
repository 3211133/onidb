# 直接採食の暫定モデリングとAgent増殖アンチパターン

植物の成長率を直接食べるクリッターを、単に「収穫重量 × 消費成長率」の収穫物消費として登録してはいけない。

## 問題

ピップのアーバー ツリー食を `木材 -26.67kg/cycle` とすると、アーバー ツリー以外が生産した木材（例: フロックス）でもピップを飼える偽経路が成立する。

この誤りは単体の質量収支だけでは発見しにくく、ループ探索で本来接続しない生産者と消費者を接続して偽陽性を作る。

## 登録規則

今回この例外処理を適用するのはピップだけとする。ほかの主体は収穫重量換算を維持する。

1. ピップ用にアーバー ツリー／シンブル リードの専用「成長量」資源を使用する。
2. 同じ植物について、通常収穫するAgentとピップ放牧に使うAgentを分ける。
3. 放牧用Agentは栽培入力から成長量だけを生産し、通常の収穫物を同時生産しない。

例:

```text
アーバー ツリー(放牧用):
  汚染水 -70, 土 -10 -> アーバー ツリー枝成長 +333.33

ピップ(アーバー ツリー):
  アーバー ツリー枝成長 -26.67 -> 土 +20
```

スキーマ追加は不要。既存の `resources` と、別レシピを別Agentにする既存方式で表現する。

## 今回採用する暫定策の問題

現行スキーマには、同一Agentが持つ複数の排他的な出力・食餌モードを表す仕組みがない。そのためピップ経路に限り、通常収穫用と放牧用を別Agentとして登録する。

これは正しい収支を得るための暫定策だが、モードやレシピの数だけAgentを増殖させるアンチパターンでもある。

- 同じ実体が複数Agentに重複する
- Agent数がループ探索の組合せ数を不必要に増やす
- 名称規則に依存し、通常収穫と放牧が同じ植物だという関係をDBが直接表現できない
- 将来の食餌追加時に登録漏れや数値の不一致が起きやすい

将来は、Agent本体と排他的なレシピ／モードを分離して表現する構造へ移行するのが望ましい。ただし今回はユーザーの指示どおりスキーマを変更せず、別Agent方式で計算を進める。
